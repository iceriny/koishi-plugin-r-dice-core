## 数据库接口 `rdiceDb`

轻量数据库服务, 只封装 `r-dice-core` 自己的数据表读写, 对命令层隐藏底层表名与 Koishi 查询细节。

- 服务文件: [../src/services/database.ts](../src/services/database.ts)
- 挂载位置: [../src/index.ts](../src/index.ts)

当前已实现的主要方法:

- `getChannel(platform, channelId)`
- `upsertChannelConfig(platform, channelId, patch)`
- `getChannelDefaultDice(platform, channelId)`
- `setChannelActiveGame(platform, channelId, gameId | null)`
- `createGame(input)`
- `getGameByName(channelPrimaryId, name)`
- `listOpenGames(channelPrimaryId)`
- `createCharacter(input)`
- `listCharacters(platform, userId)`
- `getCharacterById(id)`
- `getCharacterFields(id, fields)`
- `bindCharacterToGame(gameId, platform, userId, characterId)`
- `getBinding(gameId, platform, userId)`
- `getBindingFields(gameId, platform, userId, fields)`
- `getBoundCharacter(gameId, platform, userId)`
- `createLog(input)`
- `patchChannelById(id, patch)`
- `patchGameById(id, patch)`
- `patchBindingById(id, patch)`
- `patchCharacterById(id, patch)`

当前实现说明:

- 频道配置更新优先使用 Koishi 的 `upsert` 能力。
- 玩家与角色卡绑定使用绑定表唯一键配合 `upsert`, 避免重复绑定行。
- 开放游戏列表直接在数据库侧过滤 `archived` 状态。
- `getBoundCharacter()` 目前仍采用两段查询, 等后续出现批量读取需求再考虑切换为 `join/select`。
- 已支持字段级读取接口, 用于观察者模式下的按需加载。

### `rdiceDb` 使用示例

#### 初始化频道规则

```ts
const ruleConfig = ctx.rdiceDb.getDefaultRuleConfig("coc");

await ctx.rdiceDb.upsertChannelConfig(session.platform, session.channelId, {
  guildId: session.guildId ?? null,
  rule: ruleConfig,
});
```

#### 读取频道默认骰

```ts
const defaultDiceSides = await ctx.rdiceDb.getChannelDefaultDice(
  session.platform,
  session.channelId,
);
```

#### 按字段读取频道状态

```ts
const channel = await ctx.rdiceDb.getChannelFields(
  session.platform,
  session.channelId,
  ["id", "logOn", "activeGameId"],
);

if (channel?.logOn) {
  // 已开启日志，可以进入观察者或消息记录流程
}
```

#### 创建游戏并设置为当前游戏

```ts
const channelId = ctx.rdiceDb.createChannelPrimaryId(
  session.platform,
  session.channelId,
);

const game = await ctx.rdiceDb.createGame({
  channelId,
  name: "coc-第1团",
});

await ctx.rdiceDb.setChannelActiveGame(
  session.platform,
  session.channelId,
  game.id,
);
```

#### 绑定角色并读取当前绑定角色

```ts
await ctx.rdiceDb.bindCharacterToGame(
  game.id,
  session.platform,
  session.userId,
  character.id,
);

const currentCharacter = await ctx.rdiceDb.getBoundCharacter(
  game.id,
  session.platform,
  session.userId,
);
```

#### 创建日志记录

```ts
await ctx.rdiceDb.createLog({
  gameId: game.id,
  type: "action",
  message: ".ra 侦查",
  senderName: session.username ?? "",
  platform: session.platform,
  userId: session.userId,
  payload: {
    command: "ra",
    skill: "侦查",
  },
});
```

## 观察者服务 `rdiceObserver`

业务观察者协调服务, 只在频道已开启日志时进入完整工作模式。

- 服务文件: [../src/services/observer.ts](../src/services/observer.ts)
- 挂载位置: [../src/index.ts](../src/index.ts)

当前职责:

- 为每轮 `Session` 建立 `SessionObserverScope`
- 按频道状态判定当前会话是 `inactive`、`direct_db` 还是 `observer`
- 为 `RDiceChannel / RDiceGame / RDiceBinding / RDiceCharacter / RDiceLog` 创建自定义 observed 对象
- 记录实体已加载字段, 支持后续补读并 `$merge()`
- 在日志模式的 `middleware` 生命周期事件中统一 flush

当前激活规则:

- 未初始化频道: 不启用业务观察者
- 已初始化但未开启日志: 不启用业务观察者, 命令继续直接走 `rdiceDb`
- 已开启日志: 启用业务观察者与 middleware 统一落库

当前已实现的主要方法:

- `getScope(session)`
- `shouldUseObserver(session)`
- `observeChannel(session, fields?)`
- `observeActiveGame(session, fields?)`
- `observeBinding(session, gameId, userId?, fields?)`
- `observeCharacter(session, characterId, fields?)`
- `observeLogDraft(session, input)`
- `flushSession(session)`

当前实现说明:

- 观察者底层复用 Koishi 的 `observe()` 能力, 但生命周期由插件自己管理。
- 非日志模式不会创建业务观察者, 也不会走统一 flush。
- 插件入口不会在自身 middleware 的 `finally` 里直接落库, 而是改为监听 `ctx.on("middleware")`，避免比外层插件的后置逻辑更早提交。
- `observeLogDraft()` 当前是一次性日志草稿观察者, 在 `$update()` 时创建日志行。

### `rdiceObserver` 使用示例

#### 先判断当前会话是否进入观察者模式

```ts
if (!(await ctx.rdiceObserver.shouldUseObserver(session))) {
  return;
}
```

#### 观察当前频道与当前游戏

```ts
const channel = await ctx.rdiceObserver.observeChannel(session, [
  "id",
  "logOn",
  "activeGameId",
]);

const game = await ctx.rdiceObserver.observeActiveGame(session, [
  "id",
  "name",
  "status",
]);
```

#### 观察绑定与角色卡

```ts
const binding = await ctx.rdiceObserver.observeBinding(
  session,
  game.id,
  session.userId,
  ["id", "characterId"],
);

const character = binding
  ? await ctx.rdiceObserver.observeCharacter(session, binding.characterId, [
      "id",
      "name",
      "rule",
      "data",
    ])
  : null;
```

#### 创建日志草稿并在 flush 时统一落库

```ts
const draft = await ctx.rdiceObserver.observeLogDraft(session, {
  gameId: game.id,
  type: "action",
  message: ".ra 侦查",
  senderName: session.username ?? "",
  payload: {
    command: "ra",
  },
});

if (draft) {
  draft.message = ".ra 侦查 结果: 成功";
}

// 常规消息流程会在 `ctx.on("middleware")` 中自动 flush。
await ctx.rdiceObserver.flushSession(session);
```
