# 数据设计

本阶段只负责数据库结构设计、基于 Koishi 数据库接口的读写边界, 以及为日志场景准备的自定义观察者基础设施。不处理:

- 群消息自动转日志
- 日志聚合字段自动维护
- 非 Koishi 数据库接口的额外持久化抽象

## 设计原则

- 频道状态、游戏会话、角色卡、角色绑定、日志记录拆分为五张表。
- 频道表只保存频道自己的配置和运行状态, 不保存 `player: string[]` 这类难以表达业务语义的集合字段。
- 所有用户标识都显式带上 `platform + userId`, 避免跨平台混淆。
- 当前阶段不将 `firstLogId / lastLogId / logCount` 作为硬依赖字段, 这些都属于可推导冗余信息。
- 字段如 `status`、`type`、`rule` 的枚举语义目前主要由 TypeScript 类型与应用层校验保证, 数据库列类型仍使用 `string/json`。
- 当前实现优先使用 Koishi 的 `ctx.database.upsert()`、查询表达式与 `ctx.model.extend()` 的 `unique/indexes` 能力, 减少手写分支更新。
- 业务观察者不是全局启用, 而是按频道状态分层激活。

## RDiceChannel

频道级配置与运行状态表, 用于支撑 `.rd.init` 与 `.r` 的最小读写链路。

- `id: string`
  - 内部主键, 建议使用 `${platform}:${channelId}` 这样的稳定字符串。
- `platform: string`
  - 平台标识, 例如 `onebot`、`qq`、`telegram`。
- `channelId: string`
  - 平台内的频道或群组 ID。
- `guildId: string | null`
  - 用于兼容 `guild` 与 `channel` 分离的平台。
- `rule: { rule: string; defaultDice: number; version: string }`
  - 当前频道启用的规则配置。
- `logOn: boolean`
  - 当前频道是否正在记录日志。
- `activeGameId: number | null`
  - 当前频道正在进行中的游戏 ID。
- `createdAt: timestamp`
- `updatedAt: timestamp`

当前实现:

- 主键使用 `id`。
- 已增加联合唯一键 `(platform, channelId)`。
- 查询入口统一按 `(platform, channelId)` 定位。
- `activeGameId` 当前没有声明数据库外键。
  - 原因是 `RDiceGame.channelId -> RDiceChannel.id` 已形成一侧引用, 若频道表再直接外键指向游戏表, 会引入循环依赖。
  - 当前阶段由应用层负责维护 `activeGameId` 的一致性。

## RDiceGame

游戏或日志会话表, 用于承载某一场团的上下文状态, 是频道、日志、角色绑定之间的枢纽。

- `id: unsigned`
- `channelId: string`
  - 关联 `RDiceChannel.id`。
- `name: string`
  - 该场游戏或日志的名称。
- `status: "active" | "paused" | "archived"`
  - `active` 表示当前进行中, `paused` 表示临时关闭日志但未归档, `archived` 表示已结束归档。
- `gmPlatform: string | null`
  - GM/KP/DM 所在平台。
- `gmUserId: string | null`
  - GM/KP/DM 的平台内用户 ID。
- `createdAt: timestamp`
- `updatedAt: timestamp`
- `archivedAt: timestamp | null`

当前实现:

- `name` 不要求全局唯一, 只在频道维度内使用。
- 当前阶段不保留 `firstLogId / lastLogId / logCount`。
- 已增加索引 `(channelId, status)` 以支撑开放游戏查询。
- 已增加索引 `(channelId, name)` 以支撑按频道与名称定位游戏。

## RDiceCharacter

角色卡本体表, 与“当前在哪个团使用”解耦。

- `id: unsigned`
- `platform: string`
- `userId: string`
  - 角色所属用户的平台内 ID。
- `name: string`
- `rule: { rule: string; version: string }`
  - 角色卡所属规则与版本。
- `data: json`
  - 角色卡详细数据, 由具体规则插件解释。
- `createdAt: timestamp`
- `updatedAt: timestamp`

当前实现:

- 归属用户至少使用 `(platform, userId)` 描述。
- 是否允许同名多卡留给业务层决定, 当前阶段不写死唯一约束。

## RDiceBinding

玩家与角色卡在某场游戏中的就位关系表, 用于支撑 `.ra` 所需的 `game -> player -> character` 查询链路。

- `id: unsigned`
- `gameId: unsigned`
  - 关联 `RDiceGame.id`。
- `platform: string`
- `userId: string`
  - 当前玩家的平台内 ID。
- `characterId: unsigned`
  - 关联 `RDiceCharacter.id`。
- `createdAt: timestamp`
- `updatedAt: timestamp`

当前实现:

- 已将语义唯一键 `(gameId, platform, userId)` 落到数据库唯一约束。
- 当前阶段不增加 `isActive` 字段, 每个玩家在每个团只保留一条当前绑定。
- 绑定更新优先使用 `upsert` 语义, 避免“先查再建”带来的竞态窗口。

## RDiceLog

跑团日志表。本阶段只定义结构, 不接入消息观察者或自动记录。

- `id: unsigned`
- `gameId: unsigned`
  - 关联 `RDiceGame.id`。
- `type: "rp" | "action" | "ooc"`
  - `rp` 为角色扮演文本, `action` 为掷骰/检定等结构化动作, `ooc` 为跑团外交流。
- `message: text`
  - 原始文本。
- `payload: json`
  - 结构化数据, 替代原方案中的 `schema` 命名, 避免与数据库 schema 概念冲突。
- `senderName: string`
  - 发言时的昵称或角色名快照。
- `platform: string`
- `userId: string`
- `createdAt: timestamp`

当前实现:

- 已增加索引 `(gameId, createdAt)` 以支撑按游戏顺序拉取日志。
- 当前阶段仍只定义结构和写入入口, 不接入消息观察者。

## 基于 Koishi 数据库接口的最小服务边界

当前阶段建议只暴露轻量数据服务给命令层, 主要接口如下:

- `getChannel(platform, channelId)`
- `upsertChannelConfig(platform, channelId, patch)`
- `getChannelDefaultDice(platform, channelId)`
- `setChannelActiveGame(platform, channelId, gameId | null)`
- `createGame(input)`
- `getGameByName(channelPrimaryId, name)`
- `listOpenGames(channelPrimaryId)`
- `createCharacter(input)`
- `listCharacters(platform, userId)`
- `getCharacterById(id)`
- `getCharacterFields(id, fields)`
- `bindCharacterToGame(gameId, platform, userId, characterId)`
- `getBinding(gameId, platform, userId)`
- `getBindingFields(gameId, platform, userId, fields)`
- `getBoundCharacter(gameId, platform, userId)`
- `createLog(input)`
- `patchChannelById(id, patch)`
- `patchGameById(id, patch)`
- `patchBindingById(id, patch)`
- `patchCharacterById(id, patch)`

其中与 Koishi 数据库接口相关的当前策略为:

- `upsertChannelConfig(platform, channelId, patch)`
  - 使用 `ctx.database.upsert()` 统一处理“存在则更新, 不存在则创建”。
  - 由于 `rule` 是 JSON 字段, 当前实现会先读取现状再合并完整行, 以避免丢失未传入字段。
- `bindCharacterToGame(gameId, platform, userId, characterId)`
  - 使用 `ctx.database.upsert()` 配合绑定表唯一键 `(gameId, platform, userId)`。
  - 通过 Koishi 求值表达式保留首次 `createdAt`, 仅刷新 `updatedAt`。
- `listOpenGames(channelPrimaryId)`
  - 直接使用查询表达式在数据库侧过滤 `status != archived`, 不再把全量结果取回后在内存中过滤。
- `getBoundCharacter(gameId, platform, userId)`
  - 当前仍保留“两次查询”的直观实现。
  - 原因是当前调用路径为单条绑定读取, 还没有足够收益切换到 `join/select`；若后续出现批量角色解析, 再升级为连接查询。
- `getChannelFields()` / `getActiveGameFields()` / `getBindingFields()` / `getCharacterFields()`
  - 使用 `ctx.database.get(table, query, { fields: [...] })` 进行字段级加载。
  - 这些接口优先服务日志模式下的业务观察者, 同时也保留给普通命令路径复用。

## 自定义观察者分层激活

当前实现引入了插件侧业务观察者, 但只在 `logOn=true` 时启用:

- 未初始化频道:
  - 不创建业务观察者。
  - 命令通常直接返回“未初始化”或走各自兜底逻辑。
- 已初始化但未开启日志:
  - 不创建业务观察者。
  - 不启用业务 middleware flush。
  - `rd.init`、`r` 等命令继续直接调用 `rdiceDb` 完成数据库读写。
- 已开启日志:
  - 启用 `rdiceObserver` 服务。
  - 允许为 `RDiceChannel / RDiceGame / RDiceBinding / RDiceCharacter / RDiceLog` 建立会话级 observed 对象。
  - 在 Koishi 的 `middleware` 生命周期事件中统一调用 `$update()` 落库。

观察者模式下的当前设计:

- 每个 `Session` 使用一个 `SessionObserverScope` 管理本轮已创建的业务观察者。
- 通过字段集合记录每个实体已加载了哪些列, 支持后续补读后再 `$merge()`。
- `RDiceLog` 当前使用“日志草稿观察者”模式:
  - 本轮只在 flush 时创建一次日志行。
  - 该模式先满足日志记录入口, 不追求日志对象的二次回写。

## 接口示例

下面给出当前文档中两类核心服务的最小调用示例, 方便在命令、中间件或后续日志流程中直接复用。

### `rdiceDb` 示例

#### 初始化频道并写入规则配置

```ts
const ruleConfig = ctx.rdiceDb.getDefaultRuleConfig("dnd");

await ctx.rdiceDb.upsertChannelConfig(session.platform, session.channelId, {
  guildId: session.guildId ?? null,
  rule: ruleConfig,
});
```

#### 按字段读取当前频道状态

```ts
const channel = await ctx.rdiceDb.getChannelFields(
  session.platform,
  session.channelId,
  ["id", "logOn", "activeGameId"],
);
```

#### 读取当前游戏、绑定与角色卡

```ts
const channelId = ctx.rdiceDb.createChannelPrimaryId(
  session.platform,
  session.channelId,
);

const game = await ctx.rdiceDb.getActiveGameFields(channelId, [
  "id",
  "name",
  "status",
]);

const binding = game
  ? await ctx.rdiceDb.getBindingFields(
      game.id,
      session.platform,
      session.userId,
      ["id", "characterId"],
    )
  : null;

const character = binding
  ? await ctx.rdiceDb.getCharacterFields(binding.characterId, [
      "id",
      "name",
      "rule",
      "data",
    ])
  : null;
```

### `rdiceObserver` 示例

#### 只在日志模式启用观察者

```ts
if (!(await ctx.rdiceObserver.shouldUseObserver(session))) {
  return;
}
```

#### 建立会话级观察者并统一 flush

```ts
const channel = await ctx.rdiceObserver.observeChannel(session, [
  "id",
  "logOn",
  "activeGameId",
]);

const game = await ctx.rdiceObserver.observeActiveGame(session, [
  "id",
  "name",
  "status",
]);

const draft = game
  ? await ctx.rdiceObserver.observeLogDraft(session, {
      gameId: game.id,
      type: "ooc",
      message: "这是一条日志草稿",
      senderName: session.username ?? "",
    })
  : null;

if (draft) {
  draft.message = "这是一条已经修改过的日志草稿";
}

// 正常消息流中会由插件在 `ctx.on("middleware")` 中统一 flush。
// 只有脱离标准消息生命周期的测试或手动脚本才需要显式调用：
await ctx.rdiceObserver.flushSession(session);
```

其中本阶段先接通:

- `.rd.init` 的频道规则初始化
- `.r` 的频道默认骰读取

其他实体已具备观察者基础设施, 但只有在日志模式真正接入后才会进入完整业务流转。
